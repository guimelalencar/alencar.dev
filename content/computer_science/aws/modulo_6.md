Seção 1: Visão geral dos serviços de computação. A Amazon Web Services (AWS) oferece muitos serviços computacionais. Este é um breve resumo do que cada serviço computacional oferece: o Amazon Elastic Compute Cloud (Amazon EC2) fornece máquinas virtuais redimensionáveis; o Amazon EC2 Auto Scaling oferece suporte à disponibilidade da aplicação, permitindo que você defina condições que iniciarão ou encerrarão automaticamente instâncias do EC2; o Amazon Elastic Container Registry (Amazon ECR) é usado para armazenar e recuperar imagens de Docker; o Amazon Elastic Container Service, Amazon ECS, é um serviço de orquestração de contêineres compatível com o Docker; o VMware Cloud on AWS permite provisionar uma nuvem híbrida sem hardware personalizado; o AWS Elastic Beanstalk oferece uma maneira simples de executar e gerenciar aplicativos web; o AWS Lambda é uma solução de computação sem servidor, em que você paga apenas pelo tempo de computação usado; o Amazon Elastic Kubernetes Service (Amazon EKS) permite executar Kubernetes gerenciados na AWS; o Amazon Lightsail oferece um serviço simples de usar para criar uma aplicação ou site; o AWS Batch fornece uma ferramenta para executar trabalhos em lote em qualquer escala; o AWS Fargate oferece uma maneira de executar contêineres que reduzem a necessidade de gerenciar servidores ou clusters; o AWS Outposts oferece uma maneira de executar serviços da AWS selecionados no data center on-premises; e o AWS Serverless Application Repository permite que os clientes descubram, implantem e publiquem aplicações sem servidor. Este módulo discutirá detalhes dos serviços destacados no slide.

Imagine que cada serviço computacional da AWS pertence a uma destas quatro categorias: máquinas virtuais (VMs) que fornecem infraestrutura como serviço (IaaS), sem servidor, baseadas em contêineres e plataforma como serviço (PaaS). O Amazon EC2 fornece máquinas virtuais; imagine-o como uma infraestrutura como serviço (IaaS). Os serviços de IaaS fornecem flexibilidade e deixam muitas das responsabilidades de gerenciamento de servidores com você. Você escolhe o sistema operacional e também o tamanho e os recursos dos servidores que você inicia. Para profissionais de TI com experiência no uso de computação on-premises, as máquinas virtuais são um conceito familiar. O Amazon EC2 foi um dos primeiros serviços da AWS e continua sendo um dos serviços mais populares. O AWS Lambda é uma plataforma de computação com administração zero. O AWS Lambda permite executar código sem provisionar ou gerenciar servidores; você paga apenas pelo tempo de computação consumido. Esse conceito de tecnologia sem servidor é relativamente novo para muitos profissionais de TI; no entanto, ele está se tornando mais popular porque oferece suporte a arquiteturas nativas da nuvem, que permitem dimensionamento massivo a um custo menor do que a execução de servidores 24 horas por dia, 7 dias por semana, para oferecer suporte às mesmas cargas de trabalho.

Serviços baseados em contêiner, incluindo Amazon Elastic Container Service, Amazon Elastic Kubernetes Service, AWS Fargate e Amazon Elastic Container Registry, permitem que você execute várias cargas de trabalho em um único sistema operacional (SO). Os contêineres são mais rápidos que uma máquina virtual, portanto apresentam uma maior capacidade de resposta. As soluções baseadas em contêineres continuam a crescer em popularidade. Por fim, o AWS Elastic Beanstalk fornece uma plataforma como serviço (PaaS). Ele facilita a implantação rápida de aplicativos que você cria, fornecendo todos os serviços de aplicativos necessários. A AWS gerencia o sistema operacional, o servidor de aplicações e os outros componentes de infraestrutura para que você possa se concentrar no desenvolvimento do código do aplicativo.

A AWS oferece muitos serviços de computação porque diferentes casos de uso se beneficiam de diferentes ambientes de computação. O serviço ou serviços de computação ideais que você usa dependerão do seu caso de uso. Muitas vezes, a arquitetura de computação que você usa é determinada pelo código legado. No entanto, isso não significa que você não possa desenvolver a arquitetura para aproveitar os projetos nativos da nuvem comprovados. Algumas práticas recomendadas incluem avaliar as opções de computação disponíveis, entender as opções de configuração de computação disponíveis, coletar métricas relacionadas ao computador, usar a elasticidade disponível dos recursos e reavaliar as necessidades de computação com base em métricas. Às vezes, um cliente começa com uma solução de computação e decide alterar o design com base em sua análise de métricas. Se quiser ver um exemplo de como um cliente modificou sua escolha de serviços computacionais para um caso de uso específico, assista a este vídeo sobre solução de rastreamento de inventário em [https://www.youtube.com/watch?v=zr3Kib0i-OQ&feature=youtu.be&did=ta_card&trk=ta_card](https://www.youtube.com/watch?v=zr3Kib0i-OQ&feature=youtu.be&did=ta_card&trk=ta_card).

Apresentação da Seção 2: Amazon EC2.

A execução de servidores on-premises é uma tarefa cara. O hardware deve ser adquirido, e essa aquisição pode ser baseada em planos de projeto, em vez da realidade de como os servidores são usados. A criação, a equipe e a manutenção de data centers são caros. As organizações também precisam provisionar permanentemente uma quantidade suficiente de hardware para lidar com picos de tráfego e picos de carga de trabalho. Depois que as implantações tradicionais on-premises são criadas, a capacidade do servidor pode não ser utilizada e ficar ociosa por uma parte significativa do tempo em que os servidores estão em execução, o que é um desperdício. O Amazon Elastic Compute Cloud (Amazon EC2) fornece máquinas virtuais onde você pode hospedar os mesmos tipos de aplicações que você pode executar em um servidor on-premises tradicional. Ele disponibiliza capacidade computacional segura e redimensionável na nuvem. As instâncias do EC2 podem oferecer suporte a uma variedade de cargas de trabalho. Entre os usos comuns de instâncias do EC2 estão servidores de aplicativos, servidores web, servidores de banco de dados, servidores de jogos, servidores de e-mail, servidores de mídia, servidores de catálogo, servidores de arquivos, servidores de computação e servidores de proxy.

O EC2 no Amazon EC2 significa Elastic Compute Cloud. "Elastic" refere-se ao fato de que você pode facilmente aumentar ou diminuir o número de servidores executados para dar suporte a uma aplicação automaticamente, e também pode aumentar ou diminuir o tamanho dos servidores existentes. "Compute" refere-se ao motivo pelo qual a maioria dos usuários executa servidores, que é hospedar aplicações em execução ou processar dados, ações que exigem recursos computacionais, incluindo potência de processamento (CPU) e memória (RAM). "Cloud" refere-se ao fato de que as instâncias do EC2 que você executa estão hospedadas na nuvem.

O Amazon EC2 fornece máquinas virtuais na nuvem e controle administrativo total sobre o sistema operacional Windows ou Linux executado na instância. A maioria dos sistemas operacionais de servidores são compatíveis, incluindo Windows 2008, 2012, 2016 e 2019, Red Hat, SuSE, Ubuntu e Amazon Linux. Um sistema operacional executado em uma máquina virtual geralmente é chamado de sistema operacional convidado, para diferenciá-lo do sistema operacional host. O sistema operacional host é instalado diretamente em qualquer hardware de servidor que hospede uma ou mais máquinas virtuais. Com o Amazon EC2, é possível iniciar qualquer número de instâncias de qualquer tamanho, em qualquer Zona de Disponibilidade, em qualquer lugar do mundo, em minutos. As instâncias são iniciadas em imagens de máquinas da Amazon (AMIs), que são efetivamente modelos de máquina virtual. As AMIs são discutidas com mais detalhes posteriormente neste módulo. É possível controlar o tráfego de ida e volta para instâncias usando grupos de segurança. Além disso, como os servidores são executados na Nuvem AWS, você pode criar soluções que utilizem vários serviços da AWS.

Ao iniciar pela primeira vez uma instância do Amazon EC2, você provavelmente usará o assistente de inicialização de instância do Console de Gerenciamento da AWS. Você terá a oportunidade de testar o Launch Wizard no laboratório que está neste módulo. O Launch Instance Wizard facilita a execução de uma instância. Por exemplo, se você optar por aceitar todas as configurações padrão, pode ignorar a maioria das etapas fornecidas pelo assistente e executar uma instância do EC2 com apenas seis cliques. Um exemplo desse processo é mostrado na demonstração no final desta seção. No entanto, para a maioria das implantações, você desejará modificar as configurações padrão para que os servidores que você iniciar sejam implantados para atender às suas necessidades específicas. A próxima série de slides apresenta as escolhas essenciais que você deve fazer ao executar uma instância. Os slides abrangem conceitos essenciais que são bons para saber quando você faz essas escolhas. Esses conceitos são descritos para ajudá-lo a entender as opções que estão disponíveis e os efeitos das decisões que você tomará.

Uma imagem de máquina da Amazon (AMI) fornece informações necessárias para iniciar uma instância do EC2. Especifique uma AMI de origem ao iniciar uma instância. É possível usar AMIs diferentes para iniciar diferentes tipos de instâncias. Por exemplo, é possível escolher uma AMI para iniciar uma instância que se tornará um servidor web e outra AMI para implantar uma instância que hospedará um servidor de aplicações. Você também pode iniciar várias instâncias a partir de uma única AMI. O AMI inclui os seguintes componentes: um modelo para o volume raiz da instância, que normalmente contém um sistema operacional (SO) e tudo o que foi instalado nele (aplicações, bibliotecas etc.); execuções de permissões que controlam quais contas da AWS podem usar a AMI; e um mapeamento de dispositivo de bloco que especifica quais volumes deverão ser anexados à instância (se houver) quando ela for iniciada.

É possível escolher muitas AMIs: Quick Start, oferecidas pela AWS com opções pré-construídas de Linux e Windows; Minhas AMIs, que são as criadas por você; AWS Marketplace, que oferece um catálogo digital com milhares de soluções de software; e AMIs da comunidade, criadas por terceiros ao redor do mundo. As AMIs da comunidade não são verificadas pela AWS, portanto use-as por sua conta e risco. Elas podem oferecer muitas soluções diferentes para vários problemas, mas evite usá-las em ambientes de produção ou corporativos.

Uma AMI é criada a partir de uma instância do EC2. Você pode importar uma máquina virtual para que ela se torne uma instância do EC2 e salvar a instância como uma AMI. Em seguida, você pode iniciar uma instância do EC2 a partir dessa AMI. Como alternativa, pode começar com uma AMI existente, como as AMIs do Quick Start fornecidas pela AWS, e criar uma instância do EC2 a partir dela. Independentemente da opção escolhida, você terá o que o diagrama chama de instância não modificada. A partir dessa instância, você pode criar uma instância de ouro, ou seja, uma máquina virtual que você definiu com as configurações específicas do sistema operacional e da aplicação que deseja, e, em seguida, capturar essa instância como uma nova AMI. Quando você cria uma AMI, o Amazon EC2 interrompe a instância, cria um snapshot de seu volume raiz e registra o snapshot como uma AMI. Depois que uma AMI é registrada, ela pode ser usada para executar novas instâncias na mesma região da AWS. A nova AMI agora pode ser considerada como uma nova AMI inicial. Você também pode copiar a AMI para outras regiões, para que as instâncias do EC2 também possam ser iniciadas nesses locais.

Após escolher a AMI para iniciar a instância, escolha um tipo de instância. O Amazon EC2 fornece uma seleção de tipos de instância que podem ser otimizados para se adequarem a diferentes casos de uso. Os tipos de instância têm combinações variadas de CPU, memória, armazenamento e capacidade de rede. Os diferentes tipos de instância oferecem a flexibilidade de escolher a combinação apropriada de recursos para suas aplicações. Cada tipo de instância inclui um ou mais tamanhos de instância, permitindo que você escale os recursos de acordo com os requisitos da carga de trabalho de destino. As categorias de tipo de instância incluem instâncias de uso geral, otimizadas para computação, otimizadas para memória, otimizadas para armazenamento e computação acelerada. Cada categoria de tipo de instância oferece vários tipos de instância para escolher.

Quando você observar um tipo de instância do EC2, verá que seu nome tem várias partes. Por exemplo, considere o tipo T3. "T" é o nome da família, seguido por um número que indica a geração do tipo de instância. Portanto, uma instância t3 é a terceira geração da família T. Em geral, os tipos de instância com número de geração mais alto são mais poderosos e fornecem melhor custo-benefício. A próxima parte do nome é o tamanho da instância. Ao comparar tamanhos, é importante observar a parte do coeficiente da categoria tamanho. Por exemplo, uma t3.2xlarge tem duas vezes a vCPU e a memória de uma t3.xlarge. O t3.xlarge tem, por sua vez, duas vezes a vCPU e a memória de uma t3.large. Também é importante observar que a largura de banda da rede também está vinculada ao tamanho da instância do Amazon EC2. Se você executar trabalhos com uso intensivo de rede, pode ser necessário aumentar as especificações da instância para atender às suas necessidades.

Os tipos de instância variam de várias maneiras, incluindo tipo de CPU, contagem de CPU ou de núcleos, tipo de armazenamento, quantidade de armazenamento, quantidade de memória e desempenho de rede. O gráfico fornece uma visão de alto nível das diferentes categorias de instância e de quais famílias de tipos de instância e números de geração se adequam a cada tipo de categoria. Alguns dos tipos de instância mais detalhados incluem: instâncias T3, que fornecem instâncias de uso geral com capacidade de intermitência, adequadas para sites e aplicativos web, ambientes de desenvolvimento, servidores de compilação, repositórios de código, microsserviços, ambientes de teste e preparação e aplicações de linha de negócios; instâncias C5, otimizadas para cargas de trabalho com uso intensivo de computação, com alto desempenho e econômico a uma baixa taxa de preço por computação, usadas em modelagem científica, processamento em batch, veiculação de anúncios, jogos multijogador altamente dimensionáveis e codificação de vídeo; e instâncias R5, otimizadas para aplicativos com uso intensivo de memória, utilizadas em bancos de dados de alto desempenho, mineração e análise de dados, banco de dados em memória, cache em memória distribuído em escala da web, aplicações que executam processamento em tempo real de big data não estruturado, clusters Apache Hadoop ou Apache Spark e outros aplicativos empresariais. Para saber mais sobre cada tipo de instância, consulte a documentação do Amazon EC2: [https://aws.amazon.com/ec2/instance-types/](https://aws.amazon.com/ec2/instance-types/).

Além de considerar as necessidades de CPU, RAM e armazenamento de suas cargas de trabalho, também é importante considerar seus requisitos de largura de banda de rede. Cada tipo de instância fornece um nível de performance de rede documentado. Por exemplo, uma instância a1.medium fornecerá até 10 Gbps, mas uma instância p3dn.24xlarge fornece até 100 Gbps. Escolha um tipo de instância que atenda aos seus requisitos. Quando você executa várias novas instâncias do EC2, o Amazon EC2 tenta posicionar as instâncias para que elas sejam distribuídas pelo hardware subjacente por padrão, minimizando falhas correlacionadas. No entanto, se você quiser definir critérios específicos de posicionamento, poderá usar placement groups para influenciar o posicionamento de um grupo de instâncias interdependentes para atender às necessidades de sua carga de trabalho. Por exemplo, é possível especificar que três instâncias devem ser implantadas na mesma Zona de Disponibilidade para garantir latência de rede mais baixa e throughput de rede mais alta entre as instâncias. Consulte a documentação sobre Placement Group em [https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/placement-groups.html](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/placement-groups.html) para obter detalhes. Muitos tipos de instância também permitem configurar a rede avançada para obter um desempenho significativamente maior de pacotes por segundo (PPS), menor variação de atraso na chegada de pacotes pela rede (variação de rede) e latências mais baixas. Consulte a documentação sobre o Elastic Network Adapter (ENA) em [https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/enhanced-networking-ena.html](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/enhanced-networking-ena.html) para detalhes.

Depois de escolher uma AMI e um tipo de instância, você deve especificar o local de rede onde a instância do EC2 será implantada. A escolha de Região deve ser feita antes de iniciar o assistente de inicialização de instância. Verifique se você está na página Região correta do console do Amazon EC2 antes de escolher Iniciar instância. Quando você iniciar uma instância em uma VPC padrão, a AWS atribuirá a ela um endereço IP público por padrão. Quando você executa uma instância em uma VPC não padrão, a sub-rede tem um atributo que determina se as instâncias executadas naquela sub-rede recebem um endereço IP público do grupo de endereços IPv4 públicos. Por padrão, a AWS não atribuirá um endereço IP público a instâncias executadas em uma sub-rede não padrão. Você pode controlar se sua instância recebe um endereço IP público modificando o atributo de endereçamento IP público de sua sub-rede ou habilitando ou desabilitando o recurso de endereçamento IP público durante a execução (que substitui o atributo de endereçamento IP público da sub-rede).

É comum usar instâncias do EC2 para executar uma aplicação que deve fazer chamadas de API seguras para outros serviços AWS. Para comportar esses casos de uso, a AWS permite anexar uma função do AWS Identity and Access Management (AWS IAM) a uma instância do EC2. Sem esse recurso, você poderia ser tentado a colocar credenciais da AWS em uma instância do EC2 para que uma aplicação executada nela as use. No entanto, nunca se deve armazenar credenciais da AWS em uma instância do EC2, pois isso é altamente inseguro. Em vez disso, anexe um perfil do IAM à instância do EC2. O perfil do IAM concede permissão para fazer solicitações de API aos aplicativos que são executados na instância. Um perfil de instância é um contêiner para um perfil do IAM. Se você usar o AWS Management Console para criar uma função para o Amazon EC2, o console criará automaticamente um perfil de instância e fornecerá a ele o mesmo nome da função. Quando você usar o console do Amazon EC2 para executar uma instância com um perfil do IAM, poderá selecionar uma função para associar à instância. No console, a lista exibida é uma lista de nomes de perfis de instância. Por exemplo, um perfil do IAM pode ser usado para conceder permissões a uma aplicação executada em uma instância do EC2, como acessar um bucket no Amazon S3. É possível anexar um perfil do IAM ao executar a instância, mas também anexar uma função a uma instância do EC2 já em execução. Ao definir uma função que pode ser usada por uma instância do EC2, você define quais contas ou serviços da AWS podem assumir a função. Também define quais ações e recursos da API o aplicativo pode usar depois de assumir a função. Se você alterar uma função, a alteração será propagada para todas as instâncias que têm a função anexada.

Ao criar suas instâncias do EC2, você tem a opção de passar dados do usuário para a instância. Os dados do usuário podem automatizar a conclusão de instalações e configurações na execução da instância. Por exemplo, um script de dados do usuário pode aplicar patches e atualizar o sistema operacional da instância, buscar e instalar chaves de licença de software ou instalar software adicional. No script de dados do usuário de exemplo, há um script de shell Linux Bash simples de três linhas: a primeira linha indica que o script deve ser executado pelo shell Bash; a segunda invoca o utilitário Yellowdog Updater, Modified (YUM), usado em muitas distribuições Linux como Amazon Linux, CentOS e Red Hat Linux, para recuperar software de um repositório online e instalá-lo; a terceira linha indica que o utilitário Wget deve ser instalado. Para uma instância do Windows, o script de dados do usuário deve ser escrito em um formato compatível com prompt de comando (batch) ou Windows PowerShell. Consulte a documentação sobre scripts de dados do usuário do Windows em [https://docs.aws.amazon.com/AWSEC2/latest/WindowsGuide/ec2-windows-user-data.html](https://docs.aws.amazon.com/AWSEC2/latest/WindowsGuide/ec2-windows-user-data.html). Quando a instância do EC2 for criada, o script de dados do usuário será executado com privilégios de root durante as fases finais do processo de inicialização. Em instâncias do Linux, ele é executado pelo serviço cloud-init; em instâncias do Windows, pelo utilitário EC2Config ou EC2Launch. Por padrão, os dados do usuário são executados somente na primeira vez que a instância é iniciada. Caso deseje que o script seja executado sempre que a instância for inicializada, você pode criar um script de dados do usuário como um arquivo de várias partes no formato MIME, embora esse processo não seja comum. Mais informações estão em [https://aws.amazon.com/premiumsupport/knowledge-center/execute-user-data-ec2/](https://aws.amazon.com/premiumsupport/knowledge-center/execute-user-data-ec2/).

Ao iniciar uma instância do EC2, é possível configurar opções de armazenamento. Você pode configurar o tamanho do volume raiz onde o sistema operacional convidado está instalado e anexar volumes de armazenamento adicionais. Algumas AMIs vêm configuradas para executar mais de um volume de armazenamento por padrão, fornecendo armazenamento separado do volume raiz. Para cada volume, é possível especificar tamanho, tipo de volume, se o armazenamento será retido caso a instância seja terminada e se a criptografia será usada.

O Amazon Elastic Block Store (Amazon EBS) é um serviço de armazenamento em bloco durável, fácil de usar e de alto desempenho, projetado para cargas de trabalho com uso intensivo de transações e throughput. Com o Amazon EBS, é possível escolher entre quatro tipos de volume diferentes para equilibrar preço e desempenho ideais, e alterar tipos ou aumentar o tamanho sem interromper aplicações essenciais. O armazenamento de instância do Amazon EC2 fornece armazenamento temporário em nível de bloco para a instância, localizado em discos conectados fisicamente ao host. Funciona bem para armazenar temporariamente informações alteradas com frequência, como buffers, caches e dados temporários. Pode ser usado em grupos de instâncias replicadas, como servidores Web com balanceamento de carga. Se as instâncias forem interrompidas, dados no armazenamento de instância serão excluídos. O Amazon Elastic File System (Amazon EFS) fornece sistema de arquivo NFS elástico, dimensionável e gerenciado, usado com AWS Cloud Services e recursos on-premises, escalando sob demanda para petabytes sem interromper aplicações. O Amazon Simple Storage Service (Amazon S3) é um serviço de armazenamento de objetos com dimensionamento, disponibilidade de dados, segurança e desempenho, adequado para sites, aplicativos móveis, backup, arquivamento, IoT e análises de big data.

Dois exemplos ilustram opções de armazenamento para instâncias do EC2. No exemplo da Instância 1, o volume raiz com sistema operacional e possivelmente outros dados é armazenado no Amazon EBS, com dois volumes adicionais anexados: um volume EBS de 500 GB e um volume de armazenamento de instância. Se interrompida e reiniciada, o sistema operacional e os dados nos volumes EBS permanecerão intactos, mas dados no volume temporário serão perdidos. No exemplo da Instância 2, o volume raiz é armazenamento de instância. Uma instância com volume raiz de armazenamento de instância não pode ser interrompida via API, apenas encerrada. Se interrompida, dados do volume temporário, incluindo o sistema operacional, são perdidos. Não dependa do armazenamento de instâncias para dados de longo prazo; use Amazon EBS, Amazon EFS ou Amazon S3 para armazenamento durável. Dados no armazenamento de instância não persistirão após reinicializações.

As tags permitem categorizar recursos da AWS, como instâncias do EC2, de diferentes maneiras. Por exemplo, você pode marcar instâncias por finalidade, proprietário ou ambiente. A marcação é como anexar metadados a um recurso, permitindo pesquisa, filtragem, gerenciamento de custos e automação. Tags são especialmente úteis quando você tem várias instâncias, volumes e outros recursos, ajudando a identificar rapidamente para que cada recurso é usado e a qual projeto ou departamento ele pertence. É possível definir tags durante a criação da instância ou posteriormente, e algumas tags podem ser usadas para políticas de controle de acesso usando o AWS Identity and Access Management (IAM).

### Segurança de instâncias do EC2

O **Amazon EC2** permite controlar o tráfego de rede que entra e sai das instâncias usando **grupos de segurança** e **Listas de Controle de Acesso (ACLs) de rede**.

- **Grupos de segurança**: funcionam como firewalls virtuais. Eles controlam o tráfego de entrada e saída para instâncias específicas. As regras de grupo de segurança podem permitir ou negar protocolos, portas e endereços IP de origem. Por padrão, os grupos de segurança bloqueiam todo o tráfego de entrada, permitindo apenas o tráfego de saída. É possível anexar múltiplos grupos de segurança a uma instância.
    
- **Network ACLs (NACLs)**: aplicam regras a sub-redes inteiras, controlando o tráfego que entra e sai de todas as instâncias da sub-rede. NACLs são úteis para regras de segurança em nível de rede, complementando os grupos de segurança que operam em nível de instância.
    

Além disso, você pode usar **Elastic IPs** para fornecer endereços IP estáticos às instâncias, permitindo acesso consistente, mesmo se a instância for interrompida ou reiniciada. O Elastic IP pode ser remapeado para outra instância em caso de falha, aumentando a disponibilidade.

### Armazenamento e snapshots

O **Amazon EBS** permite criar **snapshots** de volumes, que são cópias pontuais armazenadas no Amazon S3. Snapshots podem ser usados para:

- Recuperação de dados após falha.
    
- Criar AMIs personalizadas.
    
- Migrar volumes entre regiões.
    

Snapshots são incrementais, ou seja, apenas os blocos alterados desde o último snapshot são armazenados, economizando espaço e custo.

O **armazenamento de instância** é temporário e é perdido se a instância for interrompida ou encerrada. Portanto, é indicado apenas para dados temporários, caches ou buffers, não para armazenamento persistente.

O **Amazon EFS** fornece armazenamento de arquivos escalável e compartilhado para múltiplas instâncias, ideal para aplicações que precisam de acesso simultâneo a arquivos em rede.

O **Amazon S3** oferece armazenamento de objetos altamente durável e dimensionável, útil para backup, arquivamento, análise de big data e conteúdo web.

### Balanceamento de carga e escalabilidade

Para melhorar a disponibilidade e o desempenho, o EC2 pode ser usado com **Elastic Load Balancing (ELB)**. O ELB distribui automaticamente o tráfego de entrada entre múltiplas instâncias, garantindo que nenhuma instância fique sobrecarregada. Existem três tipos de balanceadores:

- **Application Load Balancer (ALB)**: funciona no nível de aplicação (HTTP/HTTPS) e permite roteamento baseado em URL, host ou cabeçalhos.
    
- **Network Load Balancer (NLB)**: opera no nível de rede (TCP/UDP) e é ideal para alta performance e baixa latência.
    
- **Gateway Load Balancer (GLB)**: direciona tráfego de rede para appliances virtuais de terceiros, como firewalls ou IDS/IPS.
    

O **Auto Scaling** permite iniciar ou encerrar instâncias automaticamente com base em métricas, como CPU, memória ou tráfego da aplicação. Combinado com ELB, Auto Scaling garante alta disponibilidade e eficiência de custos.

### Monitoramento

O **Amazon CloudWatch** permite monitorar métricas de instâncias EC2, volumes EBS e outros recursos. É possível configurar alarmes para CPU, uso de disco, tráfego de rede e executar ações automáticas, como iniciar ou interromper instâncias, caso certas condições sejam atendidas.

O **AWS CloudTrail** registra todas as chamadas de API feitas em suas instâncias, permitindo auditoria de segurança e rastreabilidade de alterações.

Apresentação da Seção 3: otimização de custo do Amazon EC2.

A Amazon oferece diferentes modelos de preço que você pode escolher para executar instâncias do EC2. O faturamento por segundo só está disponível para instâncias sob demanda, instâncias reservadas e instâncias spot que executam o Amazon Linux ou o Ubuntu. Instâncias sob demanda estão qualificadas para o nível gratuito da AWS ([https://aws.amazon.com/free/](https://aws.amazon.com/free/)). Elas têm o menor custo inicial e a maior flexibilidade. Não há compromissos antecipados nem contratos de longo prazo. Isso é ótimo para aplicações com cargas de trabalho breves, com picos ou imprevisíveis. Os hosts dedicados são servidores físicos com a capacidade das instâncias dedicadas para seu uso. Eles permitem que você use suas licenças de software existentes por soquete, por núcleo ou por licenças do software VM, como para Microsoft Windows ou Microsoft SQL Server. Instâncias dedicadas são instâncias executadas em uma nuvem privada virtual (VPC) em hardware dedicado a um único cliente. Elas são isoladas fisicamente no nível de hardware de host das instâncias pertencentes a outras contas da AWS. A instância reservada permite que você reserve capacidade computacional por um período de 1 ou 3 anos com custos de execução por hora mais baixos. O preço de uso com desconto é fixo enquanto você é o proprietário da instância reservada. Se você espera uso consistente e pesado, eles podem fornecer economias substanciais em comparação com as instâncias sob demanda. As instâncias reservadas programadas permitem adquirir reservas de capacidade que se repetem diariamente, semanalmente ou mensalmente, com uma duração especificada, pelo período de um ano. Você paga pelo período de programação das instâncias, mesmo se não usá-las. As instâncias spot permitem que você faça sugestões de preço em instâncias do EC2 não utilizadas, o que pode reduzir seus custos significativamente. O preço por hora de uma instância spot varia de acordo com a oferta e a demanda. A instância spot é executada sempre que sua sugestão de preço excede o preço de mercado atual.

Cada modelo de preço do Amazon EC2 oferece um conjunto diferente de benefícios. As instâncias sob demanda oferecem a maior flexibilidade, sem contratos de longo prazo e taxas baixas. As instâncias spot fornecem grande escala a um preço com desconto significativo. As instâncias reservadas serão uma boa opção se você tiver necessidades de computação previsíveis ou estáveis (por exemplo, uma instância que você sabe que deseja continuar sendo executada durante meses ou anos). Os hosts dedicados são uma boa escolha quando você tem restrições de licenciamento para o software que deseja executar no Amazon EC2 ou quando você tem requisitos específicos de conformidade ou normativos que impedem o uso de outras opções de implantação.

Aqui está uma revisão de alguns casos de uso para as várias opções de preço. Os preços das instâncias sob demanda funcionam bem para cargas de trabalho com picos ou se você só precisa testar ou executar uma aplicação por um período curto (por exemplo, durante o desenvolvimento ou o teste de uma aplicação). Às vezes, suas cargas de trabalho são imprevisíveis, e as instâncias sob demanda são uma boa opção para esses casos. As instâncias spot são uma boa opção se suas aplicações puderem tolerar interrupções com uma notificação de aviso de 2 minutos. Por padrão, as instâncias são terminadas, mas é possível configurá-las para parar ou hibernar. Casos de uso comuns incluem aplicações tolerantes a falhas, como servidores web, back-ends de API e processamento de big data. As cargas de trabalho que salvam constantemente dados em armazenamento persistente (como o Amazon S3) também são boas candidatas. As instâncias reservadas são uma boa opção quando você tem cargas de trabalho de longo prazo com padrões de uso previsíveis, como servidores que você sabe que desejará executar de forma consistente ao longo de vários meses. Hosts dedicados são uma boa opção quando você tem licenças de software existentes por soquete, por núcleo ou por VM, ou quando precisa atender a requisitos específicos de conformidade corporativa e normativos.

Para otimizar custos, você deve considerar quatro motivadores consistentes e potentes: tamanho certo, aumento da elasticidade, modelo de definição de preço ideal e otimização das opções de armazenamento. Tamanho certo: escolha o equilíbrio certo dos tipos de instância. Observe quando os servidores podem ser dimensionados para baixo ou desativados e ainda atendem aos seus requisitos de desempenho. Aumento da elasticidade: projete suas implantações para reduzir a capacidade de servidor ociosa com implantações elásticas, como aquelas que usam auto scaling para lidar com picos de carga. Modelo de definição de preço ideal: reconheça as opções de definição de preço disponíveis. Analise seus padrões de uso para que você possa executar instâncias do EC2 com a combinação correta de opções de definição de preço. Otimizar opções de armazenamento: analise os requisitos de armazenamento de suas implantações. Reduza a sobrecarga de armazenamento não utilizado quando possível e escolha opções de armazenamento mais econômicas se elas ainda conseguirem atender aos seus requisitos de desempenho de armazenamento.

Primeiro, considere dimensionar corretamente. A AWS oferece aproximadamente 60 tipos e tamanhos de instância. A ampla escolha de opções permite que os clientes selecionem a instância mais adequada à sua carga de trabalho. Pode ser difícil saber por onde começar e qual opção de instância será a melhor, tanto do ponto de vista técnico quanto do custo. O tamanho correto é o processo de avaliar os recursos implantados e procurar oportunidades de redução sempre que possível. Para definir o tamanho correto: selecione a instância mais barata disponível que ainda atenda aos seus requisitos de performance; analise a utilização de CPU, RAM, armazenamento e rede para identificar instâncias que podem ser reduzidas. Você pode querer provisionar uma variedade de tipos e tamanhos de instância em um ambiente de teste e testar seu aplicativo nessas diferentes implantações de teste para identificar quais instâncias oferecem a melhor relação performance/custo. Para tamanho correto, use técnicas como testes de carga para sua vantagem. Use as métricas do Amazon CloudWatch e configure métricas personalizadas. Uma métrica representa um conjunto de valores em ordem temporal publicados no CloudWatch (por exemplo, o uso de CPU de uma instância do EC2 específica). Os pontos de dados podem ser provenientes de qualquer aplicação ou atividade de negócios para os quais você coleta dados.

Uma forma de elasticidade é criar, iniciar ou usar instâncias do EC2 quando forem necessárias, mas desativá-las quando não estiverem em uso. A elasticidade é um dos princípios centrais da nuvem, mas os clientes costumam passar por um processo de aprendizado para operacionalizar a elasticidade para gerar economias de custos. A maneira mais fácil para grandes clientes adotarem a elasticidade é procurar recursos que pareçam bons candidatos a interromper ou hibernar, como ambientes que não sejam de produção, cargas de trabalho de desenvolvimento ou cargas de trabalho de teste. Por exemplo, se você executar cargas de trabalho de desenvolvimento ou teste em um único fuso horário, poderá facilmente desativar essas instâncias fora do horário comercial e, assim, reduzir os custos de tempo de execução em, talvez, 65%. O conceito é semelhante ao porquê de haver um interruptor de luz ao lado da porta e de a maioria dos escritórios incentivar os funcionários a apagarem as luzes quando saírem do escritório todas as noites. Para cargas de trabalho de produção, configurar políticas de auto scaling mais precisas e granulares pode ajudar você a utilizar scaling horizontal para atender às necessidades de pico de capacidade e não pagar pela capacidade de pico o tempo todo. Como regra geral, você deve direcionar de 20 a 30% das suas instâncias do Amazon EC2 para serem executadas como instâncias sob demanda ou instâncias spot, e você também deve procurar ativamente formas de maximizar a elasticidade.

A AWS fornece vários modelos de preço para o Amazon EC2 para ajudar os clientes a economizar dinheiro. Os modelos disponíveis foram discutidos em detalhes anteriormente neste módulo. Os clientes podem combinar vários tipos de compra para otimizar a definição de preço com base nas necessidades de capacidade atuais e previstas. Os clientes também são incentivados a considerar a arquitetura de suas aplicações. Por exemplo, a funcionalidade fornecida pela aplicação precisa ser executada em uma máquina virtual do EC2? Talvez, ao usar o serviço AWS Lambda, você possa diminuir significativamente seus custos. O AWS Lambda é discutido mais adiante neste módulo.

Os clientes também podem reduzir os custos de armazenamento. Quando você executa instâncias do EC2, diferentes tipos de instância oferecem diferentes opções de armazenamento. É uma prática recomendada tentar reduzir custos e, ao mesmo tempo, manter o desempenho e a disponibilidade do armazenamento. Uma maneira de conseguir isso é redimensionando volumes do EBS. Por exemplo, se você provisionou originalmente um volume de 500 GB para uma instância do EC2 que só precisará de um máximo de 20 GB de espaço de armazenamento, você poderá reduzir o tamanho do volume e economizar custos. Há também uma variedade de tipos de volume do EBS. Escolha o tipo mais barato que ainda atenda aos seus requisitos de desempenho. Por exemplo, o armazenamento HDD otimizado para throughput (st1) do Amazon EBS normalmente custa metade do que a opção de armazenamento SSD de uso geral (gp2) padrão. Se uma unidade st1 atender às necessidades de sua carga de trabalho, aproveite as economias de custos. Geralmente, os clientes usam snapshots do EBS para criar backups de dados. No entanto, alguns clientes se esquecem de excluir snapshots que não são mais necessários. Exclua esses snapshots desnecessários para economizar custos. Por fim, tente identificar o destino mais apropriado para tipos específicos de dados. Seu aplicativo precisa dos dados usados para residir no Amazon EBS? Em vez disso, o aplicativo seria executado igualmente se usasse o Amazon S3 para armazenamento? A configuração de políticas de ciclo de vida de dados também pode reduzir custos. Por exemplo, você pode automatizar a migração de dados mais antigos raramente acessados para locais de armazenamento mais econômicos, como o Amazon Simple Storage Service Glacier.

Se isso for feito corretamente, a otimização de custos não é um processo único que um cliente conclui. Em vez disso, ao medir e analisar rotineiramente seus sistemas, você pode melhorar e ajustar continuamente seus custos. A marcação ajuda a fornecer informações sobre quais recursos estão sendo usados, por quem e para qual finalidade. Você pode ativar tags de alocação de custos no console Billing and Cost Management, e a AWS pode gerar um relatório de alocação de custos com uso e custos agrupados por suas tags ativas. É possível aplicar tags que representem categorias de negócios (como centros de custos, nomes de aplicativos ou proprietários) para organizar seus custos de vários serviços. Incentive as equipes a projetar custos. O AWS Cost Explorer é uma ferramenta gratuita que você pode usar para visualizar gráficos dos seus custos. É possível usar o Cost Explorer para ver padrões de despesas em recursos da AWS ao longo do tempo, identificar áreas que precisam de uma investigação mais profunda e ver tendências que você pode usar para entender os custos. Use serviços da AWS, como o AWS Trusted Advisor, que fornece orientações em tempo real para ajudar a provisionar recursos que seguem as práticas recomendadas da AWS. Os esforços de otimização de custos geralmente são mais bem-sucedidos quando a responsabilidade pela otimização de custos é atribuída a um indivíduo ou a uma equipe.

Algumas das principais lições desta seção do módulo são: os modelos de definição de preço do Amazon EC2 incluem instâncias sob demanda, instâncias reservadas, instâncias spot, instâncias dedicadas e hosts dedicados. O faturamento por segundo está disponível para instâncias sob demanda, instâncias reservadas e instâncias spot que usam apenas o Amazon Linux e o Ubuntu. As instâncias spot podem ser interrompidas com uma notificação de 2 minutos. No entanto, elas podem oferecer uma economia significativa em relação às instâncias sob demanda. Os quatro pilares da otimização de custos são: tamanho certo, aumento da elasticidade, modelo de definição de preço ideal e otimização das opções de armazenamento.

Apresentação da Seção 4: Serviços de contêiner.

Os contêineres são um método de virtualização de sistema operacional que permite executar um aplicativo e suas dependências em processos com recursos isolados. Os contêineres permitem empacotar facilmente o código, as configurações e as dependências de um aplicativo em componentes básicos fáceis de usar que oferecem consistência ambiental, eficiência operacional, produtividade de desenvolvedores e controle de versões. Os contêineres são menores do que as máquinas virtuais e não contêm um sistema operacional inteiro. Em vez disso, os contêineres compartilham um sistema operacional virtualizado e são executados como processos isolados de recursos, o que garante implantações rápidas, confiáveis e consistentes. Os contêineres mantêm tudo o que o software precisa para executar, como bibliotecas, ferramentas de sistema, código e tempo de execução. Os contêineres fornecem consistência ambiental porque o código, as configurações e as dependências do aplicativo são empacotados em um único objeto. Em termos de espaço, as imagens de contêiner geralmente são uma ordem de magnitude menor que as máquinas virtuais. Girar um contêiner acontece em centenas de milissegundos. Assim, usando contêineres, você pode usar ambientes de execução rápidos, portáteis e independentes de infraestrutura. Os contêineres podem ajudar a garantir que as aplicações sejam implantadas de forma rápida, confiável e consistente, independentemente do ambiente de implantação. Além disso, os contêineres oferecem um controle mais granular dos recursos, aumentando a eficiência da infraestrutura.

O Docker é uma plataforma de software que empacota software (como aplicativos) em contêineres. O Docker é instalado em cada servidor que hospedará contêineres e fornece comandos simples que você pode usar para criar, iniciar ou interromper contêineres. Ao usar o Docker, você pode implantar e escalar rapidamente aplicativos em qualquer ambiente. Apresentação da Seção 3: Otimização de custo do Amazon EC2.

A Amazon oferece diferentes modelos de preço que você pode escolher para executar instâncias do EC2. O faturamento por segundo só está disponível para instâncias sob demanda, instâncias reservadas e instâncias spot que executam o Amazon Linux ou o Ubuntu. Instâncias sob demanda estão qualificadas para o nível gratuito da AWS ([https://aws.amazon.com/free/](https://aws.amazon.com/free/)). Elas têm o menor custo inicial e a maior flexibilidade. Não há compromissos antecipados nem contratos de longo prazo. Isso é ótimo para aplicações com cargas de trabalho breves, com picos ou imprevisíveis.

Os hosts dedicados são servidores físicos com a capacidade das instâncias dedicadas para seu uso. Eles permitem que você use suas licenças de software existentes por soquete, por núcleo ou por licenças do software VM, como para Microsoft Windows ou Microsoft SQL Server. Instâncias dedicadas são instâncias executadas em uma nuvem privada virtual (VPC) em hardware dedicado a um único cliente. Elas são isoladas fisicamente no nível de hardware de host das instâncias pertencentes a outras contas da AWS.

A instância reservada permite que você reserve capacidade computacional por um período de 1 ou 3 anos com custos de execução por hora mais baixos. O preço de uso com desconto é fixo enquanto você é o proprietário da instância reservada. Se você espera uso consistente e pesado, eles podem fornecer economias substanciais em comparação com as instâncias sob demanda.

As instâncias reservadas programadas permitem adquirir reservas de capacidade que se repetem diariamente, semanalmente ou mensalmente, com uma duração especificada, pelo período de um ano. Você paga pelo período de programação das instâncias, mesmo se não usá-las.

As instâncias spot permitem que você faça sugestões de preço em instâncias do EC2 não utilizadas, o que pode reduzir seus custos significativamente. O preço por hora de uma instância spot varia de acordo com a oferta e a demanda. A instância spot é executada sempre que sua sugestão de preço excede o preço de mercado atual.

Cada modelo de preço do Amazon EC2 oferece um conjunto diferente de benefícios. As instâncias sob demanda oferecem a maior flexibilidade, sem contratos de longo prazo e taxas baixas. As instâncias spot fornecem grande escala a um preço com desconto significativo. As instâncias reservadas serão uma boa opção se você tiver necessidades de computação previsíveis ou estáveis (por exemplo, uma instância que você sabe que deseja continuar sendo executada durante meses ou anos). Os hosts dedicados são uma boa escolha quando você tem restrições de licenciamento para o software que deseja executar no Amazon EC2 ou quando você tem requisitos específicos de conformidade ou normativos que impedem o uso de outras opções de implantação.

Aqui está uma revisão de alguns casos de uso para as várias opções de preço. Os preços das instâncias sob demanda funcionam bem para cargas de trabalho com picos ou se você só precisa testar ou executar uma aplicação por um período curto (por exemplo, durante o desenvolvimento ou o teste de uma aplicação). Às vezes, suas cargas de trabalho são imprevisíveis, e as instâncias sob demanda são uma boa opção para esses casos.

As instâncias spot são uma boa opção se suas aplicações puderem tolerar interrupções com uma notificação de aviso de 2 minutos. Por padrão, as instâncias são terminadas, mas é possível configurá-las para parar ou hibernar. Casos de uso comuns incluem aplicações tolerantes a falhas, como servidores web, back-ends de API e processamento de big data. As cargas de trabalho que salvam constantemente dados em armazenamento persistente (como o Amazon S3) também são boas candidatas.

As instâncias reservadas são uma boa opção quando você tem cargas de trabalho de longo prazo com padrões de uso previsíveis, como servidores que você sabe que desejará executar de forma consistente ao longo de vários meses. Hosts dedicados são uma boa opção quando você tem licenças de software existentes por soquete, por núcleo ou por VM, ou quando precisa atender a requisitos específicos de conformidade corporativa e normativos.

Para otimizar custos, você deve considerar quatro motivadores consistentes e potentes:

- Tamanho certo: escolha o equilíbrio certo dos tipos de instância. Observe quando os servidores podem ser dimensionados para baixo ou desativados e ainda atendem aos seus requisitos de desempenho.
    
- Aumento da elasticidade: projete suas implantações para reduzir a capacidade de servidor ociosa com implantações elásticas, como aquelas que usam auto scaling para lidar com picos de carga.
    
- Modelo de definição de preço ideal: reconheça as opções de definição de preço disponíveis. Analise seus padrões de uso para que você possa executar instâncias do EC2 com a combinação correta de opções de definição de preço.
    
- Otimizar opções de armazenamento: analise os requisitos de armazenamento de suas implantações. Reduza a sobrecarga de armazenamento não utilizado quando possível e escolha opções de armazenamento mais econômicas se elas ainda conseguirem atender aos seus requisitos de desempenho de armazenamento.
    

Primeiro, considere dimensionar corretamente. A AWS oferece aproximadamente 60 tipos e tamanhos de instância. A ampla escolha de opções permite que os clientes selecionem a instância mais adequada à sua carga de trabalho. Pode ser difícil saber por onde começar e qual opção de instância será a melhor, tanto do ponto de vista técnico quanto do custo. O tamanho correto é o processo de avaliar os recursos implantados e procurar oportunidades de redução sempre que possível.

Para definir o tamanho correto:

- Selecione a instância mais barata disponível que ainda atenda aos seus requisitos de performance.
    
- Analise a utilização de CPU, RAM, armazenamento e rede para identificar instâncias que podem ser reduzidas. Você pode querer provisionar uma variedade de tipos e tamanhos de instância em um ambiente de teste e testar seu aplicativo nessas diferentes implantações de teste para identificar quais instâncias oferecem a melhor relação performance/custo. Para tamanho correto, use técnicas como testes de carga para sua vantagem.
    
- Use as métricas do Amazon CloudWatch e configure métricas personalizadas. Uma métrica representa um conjunto de valores em ordem temporal publicados no CloudWatch (por exemplo, o uso de CPU de uma instância do EC2 específica). Os pontos de dados podem ser provenientes de qualquer aplicação ou atividade de negócios para os quais você coleta dados.
    

Uma forma de elasticidade é criar, iniciar ou usar instâncias do EC2 quando forem necessárias, mas desativá-las quando não estiverem em uso. A elasticidade é um dos princípios centrais da nuvem, mas os clientes costumam passar por um processo de aprendizado para operacionalizar a elasticidade para gerar economias de custos. A maneira mais fácil para grandes clientes adotarem a elasticidade é procurar recursos que pareçam bons candidatos a interromper ou hibernar, como ambientes que não sejam de produção, cargas de trabalho de desenvolvimento ou cargas de trabalho de teste. Por exemplo, se você executar cargas de trabalho de desenvolvimento ou teste em um único fuso horário, poderá facilmente desativar essas instâncias fora do horário comercial e, assim, reduzir os custos de tempo de execução em, talvez, 65%. O conceito é semelhante ao porquê de haver um interruptor de luz ao lado da porta e de a maioria dos escritórios incentivar os funcionários a apagarem as luzes quando saírem do escritório todas as noites.

Para cargas de trabalho de produção, configurar políticas de auto scaling mais precisas e granulares pode ajudar você a utilizar scaling horizontal para atender às necessidades de pico de capacidade e não pagar pela capacidade de pico o tempo todo. Como regra geral, você deve direcionar de 20 a 30% das suas instâncias do Amazon EC2 para serem executadas como instâncias sob demanda ou instâncias spot, e você também deve procurar ativamente formas de maximizar a elasticidade.

A AWS fornece vários modelos de preço para o Amazon EC2 para ajudar os clientes a economizar dinheiro. Os modelos disponíveis foram discutidos em detalhes anteriormente neste módulo. É uma prática recomendada tentar reduzir custos e, ao mesmo tempo, manter o desempenho e a disponibilidade do armazenamento. Uma maneira de conseguir isso é redimensionando volumes do EBS. No entanto, alguns clientes se esquecem de excluir snapshots que não são mais necessários. Exclua esses snapshots desnecessários para economizar custos.

Por fim, tente identificar o destino mais apropriado para tipos específicos de dados. Seu aplicativo precisa dos dados usados para residir no Amazon EBS? Em vez disso, o aplicativo seria executado igualmente se usasse o Amazon S3 para armazenamento? A configuração de políticas de ciclo de vida de dados também pode reduzir custos. É possível usar o Cost Explorer para ver padrões de despesas em recursos da AWS ao longo do tempo, identificar áreas que precisam de uma investigação mais profunda e ver tendências que você pode usar para entender os custos. Use serviços da AWS, como o AWS Trusted Advisor, que fornece orientações em tempo real para ajudar a provisionar recursos que seguem as práticas recomendadas da AWS. Os esforços de otimização de custos geralmente são mais bem-sucedidos quando a responsabilidade pela otimização de custos é atribuída a um indivíduo ou a uma equipe.

Ele é ideal para organizações que já usam Kubernetes ou desejam padronizar suas operações de contêineres em múltiplos ambientes híbridos ou multi-cloud.

A escolha entre ECS e EKS depende do nível de controle e das necessidades de padronização:

- **ECS:** mais simples, totalmente integrado à AWS, fácil de gerenciar para aplicativos que não exigem funcionalidades avançadas do Kubernetes.
    
- **EKS:** oferece compatibilidade total com Kubernetes, útil se você já possui experiência com Kubernetes ou deseja portabilidade entre nuvens.
    

Em resumo, os contêineres e os serviços de contêiner da AWS oferecem:

- **Isolamento eficiente de aplicações**, com menor overhead que VMs tradicionais.
    
- **Portabilidade**, permitindo executar aplicativos em qualquer instância ou ambiente compatível com Docker.
    
- **Escalabilidade e elasticidade**, especialmente com o Fargate, sem necessidade de gerenciar infraestrutura subjacente.
    
- **Integração com outros serviços AWS**, como ELB, CloudWatch e IAM.
    
- **Otimização de custos**, permitindo usar instâncias spot, reduzir servidores ociosos e ajustar recursos conforme a demanda.
    

Esses serviços são particularmente úteis para arquiteturas de **microserviços**, aplicações com **cargas variáveis** e equipes que desejam acelerar o desenvolvimento e a entrega contínua de software.

Seção 5: AWS Lambda e Computação Sem Servidor

O **AWS Lambda** é um serviço de computação sem servidor que permite executar código sem precisar provisionar ou gerenciar servidores. Com o Lambda, você só paga pelo tempo de execução do seu código, medido em **milissegundos**, e pelo número de solicitações, sem custos com instâncias ociosas. Isso reduz significativamente a sobrecarga operacional e financeira.

O Lambda funciona com **funções**, que são blocos de código que executam tarefas específicas. Cada função é configurada com:

- Linguagem de programação suportada (como Python, Node.js, Java, Go, C#).
    
- Memória alocada (que determina também a CPU proporcional).
    
- Timeout máximo de execução (até 15 minutos por invocação).
    

Uma **invocação** ocorre quando um evento aciona a execução de uma função. Os eventos podem vir de múltiplos serviços da AWS, como:

- **Amazon S3:** acionando funções quando objetos são criados, modificados ou excluídos.
    
- **Amazon DynamoDB:** respondendo a alterações em tabelas (streams).
    
- **Amazon API Gateway:** executando funções em resposta a chamadas de APIs REST ou HTTP.
    
- **Amazon CloudWatch Events ou EventBridge:** agendando funções em horários específicos ou reagindo a eventos do sistema.
    
- **SNS (Simple Notification Service):** enviando notificações que acionam funções Lambda.
    

O Lambda escala automaticamente: quando vários eventos ocorrem simultaneamente, novas instâncias da função são criadas para atender à demanda. Isso elimina a necessidade de configurar **auto scaling** manualmente.

**Principais características do AWS Lambda:**

- **Modelo de preço baseado em uso:** você paga apenas pelo tempo de execução e número de solicitações.
    
- **Execução sem servidor:** sem necessidade de gerenciar máquinas virtuais ou clusters.
    
- **Escalabilidade automática:** o serviço ajusta o número de instâncias da função conforme a demanda.
    
- **Integração com outros serviços AWS:** permite automatizar fluxos de trabalho e arquiteturas de microsserviços.
    
- **Execução de código em resposta a eventos:** ideal para pipelines de dados, processamento em tempo real, e tarefas programadas.
    

**Casos de uso comuns para AWS Lambda:**

- Processamento de arquivos enviados ao S3, como redimensionamento de imagens ou conversão de formatos.
    
- Criação de APIs sem servidor usando API Gateway e Lambda.
    
- Automatização de processos internos, como limpeza de logs ou sincronização de bancos de dados.
    
- Processamento de streams de dados em tempo real usando DynamoDB Streams ou Kinesis.
    
- Orquestração de tarefas entre múltiplos serviços AWS usando Step Functions.
    

**Integração com outras soluções sem servidor:**

- **AWS Step Functions:** permite orquestrar funções Lambda em fluxos de trabalho complexos, definindo sequências, paralelismo e tratamento de falhas.
    
- **Amazon EventBridge:** conecta eventos de fontes internas ou externas e dispara funções Lambda para automatizar respostas.
    
- **Amazon S3 e DynamoDB:** processa dados automaticamente quando alterações ocorrem, garantindo respostas quase em tempo real.
    

**Vantagens da computação sem servidor com Lambda:**

- Redução de custo e complexidade, eliminando servidores ociosos.
    
- Escalabilidade automática sem necessidade de intervenção manual.
    
- Rapidez na implantação e alteração de funções, permitindo entrega contínua de software.
    
- Capacidade de criar arquiteturas orientadas a eventos, altamente reativas e flexíveis.
    

**Boas práticas para AWS Lambda:**

- Manter funções pequenas e focadas em uma única responsabilidade.
    
- Reutilizar bibliotecas externas com camadas (Lambda Layers) para reduzir o tamanho das funções.
    
- Configurar corretamente o timeout e a memória, equilibrando custo e desempenho.
    
- Monitorar funções com Amazon CloudWatch para capturar logs e métricas de desempenho.
    
- Testar e versionar funções para garantir estabilidade e rastreabilidade.
    

O Lambda é ideal para **aplicações sem servidor**, automação de processos e arquiteturas baseadas em eventos, permitindo que equipes foquem no código e na lógica de negócio sem se preocupar com infraestrutura subjacente.

Apresentação da Seção 6: Introdução ao AWS Elastic Beanstalk.

O AWS Elastic Beanstalk é outra opção de serviço computacional da AWS. É uma plataforma como um serviço (ou PaaS) que facilita a rápida implantação, escalabilidade e gerenciamento de suas aplicações e serviços web. Você permanece no controle. Toda a plataforma já está criada e você só precisa fazer upload do código. Escolha seu tipo de instância, seu banco de dados, defina e ajuste a auto scaling, atualize seu aplicativo, acesse os arquivos de log do servidor e habilite HTTPS no balanceador de carga. Você faz o upload de seu código e o Elastic Beanstalk gerencia automaticamente a implantação, desde o provisionamento de capacidade, balanceamento de carga e auto scaling até o monitoramento da integridade de aplicativos. Ao mesmo tempo, você mantém total controle sobre os recursos da AWS que possibilitam a operação da aplicação e acessa os recursos subjacentes a qualquer momento. Não há custo adicional para o AWS Elastic Beanstalk. Você paga pelos recursos da AWS (como instâncias do EC2 ou buckets do S3) que criar para armazenar e executar seu aplicativo. Você só paga pelo que usa, no momento em que usa. Não há taxas mínimas nem compromissos antecipados.

O AWS Elastic Beanstalk permite implantar código por meio do Console de Gerenciamento da AWS, da Interface da Linha de Comando da AWS (CLI da AWS), do Visual Studio e do Eclipse. Ele fornece todos os serviços de aplicativos de que você precisa para seu aplicativo. A única coisa que você deve criar é seu código. O Elastic Beanstalk foi projetado para tornar a implantação de seu aplicativo um processo rápido e fácil. O Elastic Beanstalk oferece suporte a uma grande variedade de plataformas. As plataformas compatíveis incluem Docker, Go, Java, .NET, Node.js, PHP, Python e Ruby. O AWS Elastic Beanstalk implanta seu código no Apache Tomcat para aplicações Java; no Apache HTTP Server para aplicações PHP e Python; no NGINX ou no Apache HTTP Server para aplicações Node.js; no Passenger ou no Puma para aplicações Ruby e no Microsoft Internet Information Services (IIS) para aplicações .NET, Java SE, Docker e Go.

O Elastic Beanstalk é rápido e simples de começar a usar. Use o Console de Gerenciamento da AWS, um repositório Git ou um ambiente de desenvolvimento integrado (IDE) como o Eclipse ou o Visual Studio, para fazer upload de sua aplicação. O Elastic Beanstalk trata automaticamente os detalhes da capacidade de provisionamento, balanceamento de carga, auto scaling e monitoramento de integridade da aplicação. Você pode melhorar a produtividade do desenvolvedor ao se concentrar em escrever código em vez de gerenciar e configurar servidores, bancos de dados, balanceadores de carga, firewalls e redes. A AWS atualiza a plataforma subjacente que executa sua aplicação com patches e atualizações. O Elastic Beanstalk é difícil de superar. Com o Elastic Beanstalk, sua aplicação pode lidar com picos de carga de trabalho ou tráfego ao minimizar os custos. Ele aumenta ou diminui automaticamente o aplicativo com base nas necessidades específicas do aplicativo usando configurações de auto scaling facilmente ajustáveis. Você pode usar métricas de utilização da CPU para disparar ações de auto scaling. Você tem a liberdade de selecionar os recursos da AWS, como o tipo de instância do Amazon EC2, ideais para a sua aplicação. O Elastic Beanstalk permite que você mantenha controle total sobre os recursos da AWS que capacitam sua aplicação. Se você optar por assumir o controle de alguns (ou todos) os elementos da sua infraestrutura, poderá fazê-lo com perfeição usando os recursos de gerenciamento do Elastic Beanstalk.

Algumas das principais lições desta seção do módulo são: o AWS Elastic Beanstalk aprimora a produtividade do desenvolvedor; simplifica o processo de implantação da aplicação; reduz a complexidade do gerenciamento; o Elastic Beanstalk é compatível com Java, .NET, PHP, Node.js, Python, Ruby, Go e Docker; o Elastic Beanstalk não é cobrado. Pague apenas pelos recursos da AWS que você usar.
